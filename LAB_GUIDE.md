# Week 13: Advanced RAG Techniques - Bootcamp Lab Guide

Welcome to the Week 13 Bootcamp Lab! In this module, we will explore **Advanced Retrieval-Augmented Generation (RAG) Techniques**. Simple RAG (which you learned in Week 12) often suffers from precision and recall issues when dealing with complex or ambiguous queries. 

To solve this, we are introducing three powerful advanced strategies in this project:
1. **Query Expansion:** Automatically generating multiple variations of a user's question to maximize search coverage.
2. **Parallel Search:** Running multiple vector searches concurrently to fetch the best documents for every query variation.
3. **LLM Re-Ranking:** Using a Large Language Model to act as a "judge" to score, evaluate, and perfectly re-rank documents based on their actual relevance to the query.

## 3. Lab Part 2: Query Expansion & Parallel Search

In base RAG, a user searching for *"How to connect to DB"* might miss documents that say *"Database connection guide"*. **Query Expansion** fixes this by asking the LLM to rewrite the query into multiple variations (e.g., *"Database connection guide"*, *"PostgreSQL connection steps"*).

**Parallel Search** takes those variations and searches PgVector *simultaneously* for all of them, dramatically improving recall.

### 3.1 The Search Endpoint
**Endpoint:** `POST /api/rag/search`

**Task:** Test the standard advanced search pipeline.
```bash
curl -X POST http://localhost:8080/api/rag/search \
  -H "Content-Type: application/json" \
  -d '{
    "query": "Tell me about AI architecture optimization",
    "filters": {}
  }'
```

**What happens behind the scenes?**
1. `QueryExpansionService` asks the LLM to generate 3-5 variations of your query.
2. `ParallelSearchService` executes vector similarity searches for each variation **in parallel**.
3. It aggregates the results, removes duplicates, and returns them to you.

You will see `queryVariations` generated by the LLM in your JSON response!

## 4. Lab Part 3: LLM Re-Ranking (The "Judge")

When you use Query Expansion, you might fetch *too many* documents, including irrelevant ones. The vector database only understands mathematical similarity, not deep logical context.

**LLM Re-Ranking** takes the "shuffled" documents from the parallel search and passes them back to the LLM. The LLM acts as an intelligent judge, scores each document based on its strict relevance to the original query, and sorts them from highest to lowest.

### 4.1 The LLM Re-Ranker Endpoint
**Endpoint:** `POST /api/rag/search-llm`

**Task:** Query the API using the LLM Re-ranking engine.
```bash
curl -X POST http://localhost:8080/api/rag/search-llm \
  -H "Content-Type: application/json" \
  -d '{
    "query": "Tell me about AI architecture optimization",
    "filters": {}
  }'
```

**Architecture Flow for `/search-llm`:**
1. `QueryExpansionService`: Original query -> N query variations
2. `ParallelSearchService`: Search PgVector for all N variations -> Initial Results List
3. `LLMReRankerService`:
   - Takes the Initial Results.
   - Puts them into a prompt: *"Score these documents against the user's query."*
   - Extracts the LLM-assigned scores.
   - Re-sorts the documents.
4. Returns the highly precise, re-ranked Results List!
